# FallGuard 项目规范

FallGuard 是一个用于课设演示的摔倒检测项目。项目核心不是单纯的 GUI，而是使用预训练 YOLO-Pose 提取人体框和姿态关键点，再基于姿态特征与短时序特征判断当前基础状态，最后通过 Streamlit 展示监测画面、状态和历史事件。

## 项目目标

- 使用 Python 完成全部实现。
- 支持本地视频文件回放和电脑摄像头实时测试。
- 使用预训练 YOLO-Pose 获取人体框和姿态关键点。
- 基于人体姿态特征判断基础状态：`unknown`、`standing`、`sitting`、`moving`、`lying`、`fall`。
- 能区分主动躺下和摔倒：慢速、平稳进入低姿态判为 `lying`；快速下降、姿态突变并持续低姿态判为 `fall`。
- 使用 Streamlit 构建后台演示平台，展示标注视频、当前状态和历史事件。

## 技术约束

- 项目语言限定为 Python。
- 后台演示平台使用 Streamlit。
- YOLO-Pose 只负责人体框和姿态关键点提取，不直接输出站立、坐下、躺下或摔倒状态。
- 不把摔倒作为 YOLO 的独立检测类别。
- 当前阶段聚焦预训练 YOLO-Pose、姿态特征判断和 Streamlit 演示平台。
- 状态判断应由统一的姿态特征模块完成，避免把多套互相冲突的判断规则混在一起。

## 参考思路

项目实现可参考以下公开方向，但需要服务于本项目的 Python 演示目标：

- YOLO-Pose / OpenPose：使用人体关键点表达人体姿态。
- UP-Fall、UR Fall Detection Dataset、Le2i、MCFD：参考其中的摔倒和日常活动场景，设计本地测试视频和状态判断边界。
- 基于姿态关键点的摔倒检测论文：参考身体倾角、髋部高度变化、人体水平程度、低姿态持续时间等特征。

这些资料用于指导状态定义、特征选择和测试场景，不在本文件中展开训练路线。

## 状态定义

状态输出只保留基础状态，便于课设演示和后续实现：

- `unknown`：人体未被可靠检测，或关键点置信度不足。
- `standing`：人体整体接近竖直，髋部位置较高，肩髋方向稳定。
- `sitting`：人体高度下降，膝髋关系符合坐姿，身体未完全水平。
- `moving`：人体处于正常移动过程，姿态变化存在但没有快速跌落特征。
- `lying`：人体处于低姿态或接近水平，进入过程较慢、平稳，可视为正常躺下。
- `fall`：人体快速下降或姿态突变后进入低姿态，并在短时间内没有恢复。

## 姿态特征判断

状态判断模块应接收连续帧中的人体框、关键点和关键点置信度，输出当前基础状态。

单帧特征建议包括：

- 人体框位置、宽高和外接框长宽比。
- 关键点整体置信度。
- 肩、髋、膝、踝等关键点的相对位置。
- 身体主轴倾角，例如肩部中心到髋部中心的方向。
- 人体水平程度，例如肩髋连线、髋膝踝分布和外接框比例。

短时序特征建议包括：

- 髋部或人体中心高度变化。
- 人体中心下降速度。
- 身体倾角变化速度。
- 低姿态持续时间。
- 进入低姿态后是否快速恢复。

外接框长宽比可以作为姿态特征之一，但不能单独决定是否摔倒。实现时应优先结合关键点位置、人体中心高度变化、姿态变化速度和低姿态持续时间。

## 主动躺下与摔倒区分

主动躺下和摔倒的最终画面都可能接近低姿态，因此不能只看单帧结果。

- 主动躺下通常表现为：下降速度较慢、姿态变化较平滑、进入低姿态过程可控，最终状态判为 `lying`。
- 摔倒通常表现为：人体中心快速下降、身体倾角快速变化、短时间内进入低姿态，且没有快速恢复，最终状态判为 `fall`。
- 当关键点置信度低、遮挡严重或只检测到局部人体时，应优先输出 `unknown`，避免强行判断。

## 可视化输出

视频画面必须支持同屏显示以下内容：

- 人体框。
- 姿态关键点。
- 关键点之间的骨架连线。
- 状态文字，例如 `standing`、`lying`、`fall`。

状态文字应来自姿态特征判断模块，而不是 YOLO-Pose 原始输出。人体框和骨架线用于解释状态判断依据，适合课设演示和错误分析。

## Streamlit 后台平台

后台平台用于演示本地监测结果，至少包含：

- 视频文件输入或摄像头输入选择。
- 标注后的视频画面。
- 当前状态显示。
- 历史事件列表，包括时间、状态和简要特征摘要。
- 摔倒状态的明显提示。

平台只负责展示和交互，不应承担姿态估计或状态判断的核心逻辑。核心逻辑应放在独立 Python 模块中，方便命令行测试和后续复用。

## 隐私安全方案

默认隐私原则是视频在本地处理，不上传原始视频。

- 姿态估计和状态判断在本地电脑完成。
- 后台只展示处理后的标注画面、状态和事件历史。
- 事件记录中优先保存时间、状态、置信度或特征摘要。
- 如需保留画面用于课设展示，应只保留必要样例，并避免包含无关人员、敏感场景或可识别隐私信息。

这种方案可以在开发阶段用本地电脑模拟摄像头边缘端；后续迁移到边缘设备时，只需要替换视频输入和运行环境，核心状态判断逻辑保持一致。

## 建议目录结构

可按以下职责组织代码：

- `src/pose/`：YOLO-Pose 推理与关键点解析。
- `src/features/`：单帧姿态特征和短时序特征计算。
- `src/state/`：基础状态判断。
- `src/visualization/`：人体框、骨架线和状态标签绘制。
- `app/`：Streamlit 后台演示页面。
- `tests/`：姿态特征、状态判断和可视化逻辑测试。
- `data/samples/`：本地测试视频或摄像头测试说明。

目录结构可以根据实际实现微调，但模块职责应保持清晰。

## 验收标准

- 可以通过视频文件或电脑摄像头运行演示。
- 系统能输出 `unknown`、`standing`、`sitting`、`moving`、`lying`、`fall` 中的基础状态。
- 系统能用短时序特征区分主动躺下和摔倒。
- 视频画面能同时显示人体框、姿态关键点、骨架连线和状态文字。
- Streamlit 页面能展示标注画面、当前状态和历史事件。
- 文档和实现都应保持 Python-only，不引入前端框架或非 Python 平台作为主方案。
